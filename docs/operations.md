# Operations Manual — motto-common

## Service Level Indicators (SLIs)

motto-common is a foundational Python shared package consumed by all Motto fleet
repositories.  The indicators below measure the health of the package from a
consumer's perspective.

| SLI | What It Measures | How It Is Measured |
|-----|-----------------|--------------------|
| **Install success rate** | Percentage of `uv add motto-common` or `pip install motto-common` attempts that complete without errors. | CI matrix installs the package on Python 3.12, 3.13, and 3.14 in the test workflow; any failure counts against the rate. |
| **Import time** | Wall-clock time for `import motto_common` (cold import, no cached .pyc). | Measured by the `test_import_time_is_reasonable` integration test; a subprocess import is timed via `time.perf_counter()`. |
| **Test pass rate** | Percentage of pytest test cases that pass across the full unit + integration suite. | `uv run pytest tests/ integration_tests/ -q` exit code and summary line. |
| **Lint & type-check pass rate** | Whether ruff and mypy report zero errors against the latest commit. | CI pre-commit job runs ruff + mypy; any non-zero exit is a failure. |
| **Security scan pass rate** | Whether Bandit reports zero high/medium severity findings. | CI bandit job runs `bandit -c pyproject.toml -r src/`. |

## Service Level Objectives (SLOs)

The table below sets quantified targets for each SLI.  These are internal
health targets for the motto-common maintainers.

| SLO | Target | Measurement Window |
|-----|--------|--------------------|
| Install success rate | ≥ 99.5% | Rolling 30 days (all CI runs) |
| Import time (cold, p50) | ≤ 200 ms | Every CI run (Python 3.12) |
| Import time (cold, p99) | ≤ 500 ms | Every CI run (Python 3.12) |
| Test pass rate | = 100% | Per commit (any test failure blocks merge) |
| Lint / type-check compliance | = 100% | Per commit (any violation blocks merge) |
| Bandit security findings | Zero high/medium | Every CI run |

### SLO Rationale

- **Install success rate ≥ 99.5%** — As a foundational dependency, install
  failures cascade to every consumer.  The target accounts for transient PyPI
  / network issues while requiring near-perfect packaging.
- **Import time ≤ 200 ms** — motto-common is imported at cold start by every
  Motto agent.  A fast import keeps fleet startup latency low.
- **Test pass rate = 100%** — Every test failure is treated as blocking.
  There is zero tolerance for regressions in a shared package consumed by
  18 repos.
- **Lint / type-check = 100%** — Enforced by pre-commit hooks and CI.
  Allows zero warnings.
- **Bandit zero high/medium** — motto-common makes limited system calls
  (`subprocess` for git SHA).  These are audited and annotated with
  `# nosec` where justified.

## Runbooks

### Dependency Updates

#### Routine Update (minor/patch)

1. Create a feature branch:
   ```bash
   git checkout -b deps/update-<package>-<version>
   ```
2. Update the dependency:
   ```bash
   uv add <package>           # for runtime deps
   uv add --dev <package>     # for dev deps
   uv lock --upgrade-package <package>
   ```
3. Run the full verification suite locally:
   ```bash
   uv run ruff check .
   uv run mypy .
   uv run bandit -c pyproject.toml -r src/
   uv run pytest tests/ integration_tests/ -v
   ```
4. Review `uv.lock` diff for unexpected transitive changes.
5. Open a PR.  All CI checks must pass before merging.

#### Major Version Update

1. Follow the routine update steps above.
2. Check the upstream changelog for breaking changes.
3. If the dependency is `sentry-sdk`, manually verify Sentry event delivery
   in a staging deployment of one consumer (e.g., motto-director).
4. If the dependency is a build tool (hatchling, ruff, mypy), update the
   version constraint in `pyproject.toml` `[build-system].requires` or
   `[dependency-groups].dev` as appropriate.

#### Security Patch (Dependabot)

1. Dependabot will automatically open PRs for known vulnerabilities.
2. Review the advisory linked in the PR description.
3. Merge if CI passes.  For critical CVEs, expedite by bypassing the full
   test matrix (use admin override on the PR).

### Release Process

#### Pre-release Checklist

1. Ensure `main` branch is green: all CI jobs passing.
2. Review `CHANGELOG.md` for accuracy (auto-generated by semantic-release).
3. Decide the version bump:
   - **Patch** (`v0.1.x`): bug fixes, docs, non-functional changes
   - **Minor** (`v0.x.0`): new backwards-compatible functionality
   - **Major** (`vX.0.0`): breaking changes (rare for shared lib)
4. Verify PyPI Trusted Publishing is configured (see `docs/pypi-trusted-publishing.md`).

#### Creating a Release

1. Tag the release on the merge commit:
   ```bash
   git checkout main
   git pull origin main
   git tag v0.1.0   # use the actual version
   git push origin v0.1.0
   ```
2. The `release.yml` workflow triggers automatically on `v*` tag push:
   - Runs semantic-release to update CHANGELOG.md
   - Builds the wheel with `uv build`
   - Publishes to PyPI via Trusted Publishing (OIDC)
   - Creates a GitHub Release with auto-generated notes
3. Monitor the [Release workflow run](https://github.com/lkmotto/motto-common/actions/workflows/release.yml).
4. After publish succeeds, verify on PyPI:
   ```bash
   uv add "motto-common==<new-version>"
   python -c "from motto_common.sentry_init import init_sentry; print('OK')"
   ```

#### Post-release

1. Verify that at least one consumer (e.g., motto-director) can install the
   new version and pass its test suite.
2. If the release introduced new features, update the consumer repos'
   dependencies to reference the new version.
3. If the release was a hotfix, notify the fleet via the Motto Slack channel.

### Incident Response

#### Symptom: Consumers failing to install motto-common

1. Check PyPI status at https://status.pypi.org/.
2. Verify the package exists: `uv pip install motto-common`.
3. Check if a recent release introduced a packaging regression:
   - Look at the latest tag: `git tag --sort=-version:refname | head -1`
   - Check CI for that tag's release workflow run.
4. If the package is corrupted, yank the release on PyPI:
   ```bash
   # Via PyPI web UI or twine
   # Then push a fix release.
   ```

#### Symptom: Import time regression

1. The `test_import_time_is_reasonable` integration test will catch this.
2. Profile the import:
   ```bash
   python -X importtime -c "import motto_common" 2>&1 | sort -rn | head -20
   ```
3. Check if a recent dependency update introduced heavy imports.
4. Consider lazy-loading non-critical modules.

#### Symptom: Sentry not receiving events from a consumer

1. Verify the consumer is setting `SENTRY_DSN` in its environment.
2. Check that `init_sentry(agent_name="...")` is called with the correct
   agent name.
3. Verify the Sentry project exists and the DSN is valid.
4. Check the consumer's logs for Sentry SDK warnings (the SDK logs
   connectivity issues at WARNING level by default).

## CI/CD Pipeline

The test workflow (`.github/workflows/test.yml`) runs:

| Trigger | What Runs |
|---------|-----------|
| Push to `main` / `master` | Full matrix: pre-commit, bandit, test (3.12, 3.13, 3.14) |
| Pull request to `main` / `master` | Same as push |
| Nightly cron (`0 6 * * *`) | Full matrix + integration tests + all security scans |

The nightly cron ensures we catch regressions from upstream dependency changes
(even when no code changes are pushed to motto-common) and provides a reliable
daily health check.
